# Phase 02: CI/CD Setup

[![Runbook](https://img.shields.io/badge/Type-Runbook-blue?style=flat-square)](../index.mdx)
[![GitLab](https://img.shields.io/badge/Platform-GitHub-181717?style=flat-square&logo=github)](https://gitlab.com)

> **DOCUMENT CATEGORY**: Stage Overview 
> **SCOPE**: source control project creation and CI/CD configuration 
> **PURPOSE**: Establish customer source control project with proper settings and environments 
> **MASTER REFERENCE**: [CI/CD pipeline Documentation](https://docs.gitlab.com/ee/ci/)

**Status**: Active 
**Applies To**: All Azure Local deployments 
**Last Updated**: 2026-01-31

---

## Stage Purpose

Create customer source control project from Azure Local Cloud template, configure project settings for secure deployments, create GitLab environments, and configure environment-specific variables. This stage establishes the CI/CD foundation for infrastructure automation.

---

## Prerequisites

Before starting this stage:

- [ ] Stage 00 (B2B Configuration) completed
- [ ] GitLab access with project creation permissions
- [ ] Customer information (CID, name, service tier)
- [ ] Access to source control templates

| Step | Title | Description |

## Key Deliverables

| Deliverable | Description |
|-------------|-------------|
| **source control project** | Customer project created from Azure Local Cloud template |
| **Project Naming** | Follows convention: `cid###-customer_name` |
| **Branch Protection** | Main branch protected with merge request requirements |
| **Merge Approvals** | Configured approval rules for code review |
| **Security Settings** | Push rules, security scanning enabled |
| **Environments** | GitLab environments for each deployment target |
| **Variables** | Environment-specific variables configured |

---

## Project Naming Convention

| Component | Format | Example |
|-----------|--------|---------|
| **Customer ID** | `cid###` | `cid123` |
| **Customer Name** | lowercase, underscores | `Infinite azurelocal Corp_corp` |
| **Full Name** | `cid###-customer_name` | `cid123-Infinite azurelocal Corp_corp` |

---

## Service Tier Folders

Projects are placed in folders based on service tier:

| Folder | Service Tier |
|--------|--------------|
| `customers/premium/` | Public Cloud Premium |
| `customers/professional/` | Public Cloud Professional |
| `customers/megaport/` | Megaport customers |

---

## GitLab Environments

Standard environments created for each customer:

| Environment | Description | Use Case |
|-------------|-------------|----------|
| `azr-prod-us` | Production US region | Primary production deployments |
| `azr-dev-us` | Development US region | Development/testing |
| `azr-prod-eu` | Production EU region | European deployments (if applicable) |
| `azr-staging` | Staging environment | Pre-production validation |

---

## Project Settings Checklist

### Branch Protection
- [ ] Main branch protected
- [ ] Force push disabled
- [ ] Deletion disabled
- [ ] Merge requests required

### Merge Request Settings
- [ ] Approvals required before merge
- [ ] All discussions must be resolved
- [ ] Semi-linear history enforced
- [ ] Pipelines must succeed

### Security Settings
- [ ] Push rules configured
- [ ] Secret detection enabled
- [ ] SAST scanning enabled
- [ ] Dependency scanning enabled

---

## Validation Checklist

- [ ] source control project created with correct naming
- [ ] Project placed in correct service tier folder
- [ ] Branch protection rules applied
- [ ] Merge request approvals configured
- [ ] Push rules enabled
- [ ] Environments created
- [ ] Environment variables configured
- [ ] Test pipeline runs successfully
